# 审查工作流

## 审查模式规则

- 审查模式下不编辑文件。
- 读取足够多的上下文代码，以验证每条发现。
- 优先关注具体缺陷，而非泛泛的风格建议。
- 除非某条 `python-standards.md` 中的偏好设置会造成真实的可维护性、正确性、可靠性或运营风险，否则不将其上报为 bug。

## 发现优先级

按严重程度排序，优先列出以下类型的发现：

- 正确性 bug 和行为回归。
- 数据丢失、安全、权限或破坏性操作风险。
- 缺失或薄弱的错误处理（可能掩盖故障）。
- 高代价 I/O、计算、内存使用或提前加载导致的性能风险。
- 对已改动行为有重要意义的测试缺口。

## 发现格式

- 尽可能附上受影响的文件和精确的行号引用。
- 说明用户可见的或运营层面的影响。
- 说明触发条件或代码路径。
- 修复方向不明显时，给出简洁的修复建议。
- 尽可能用 `file:line` 格式标注每条发现。

## 应避免的行为

- 审查模式下不改写解决方案。
- 不罗列所有细微的风格问题。
- 在没有风险证据的情况下，不将缺失的偏好设置视为缺陷。
- 不在列出发现前大量称赞或总结。

## 如果没有发现问题

- 明确说明未发现问题。
- 提及任何残余风险、未测试路径，或未运行的验证项。
- 保持简洁。

## 类型审查（温和模式）

审查涉及类型标注的代码时，采用温和立场：

- **鼓励修复优于抑制**：发现 `# type: ignore` 时，优先建议添加类型标注、安装 stub 包、使用 isinstance 收窄。抑制是最后手段。
- **允许有理由的 `# type: ignore`**：`# type: ignore[import-untyped]` 在第三方库无 stub 时可接受，但需附理由注释。裸 `# type: ignore` 不接受。
- **允许附上下文的 assert**：`assert x is not None, "post-validation guarantee"` 可接受。裸 `assert x is not None` 建议改为早返回或 raise。
- **标记 `Any` 并建议替代**：发现 `Any` 时建议 `object` 或更具体的类型，但不强制。
- **修复优先级参考**：类型问题的严重程度参考 `typing-guide.md` 的 6 层优先级（仅当本次任务涉及类型审查时，按入口路由读取）——低层问题（如缺失返回类型）比高层问题（如 TypeVar 协变）更值得关注。
