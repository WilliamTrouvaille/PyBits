# 写代码工作流

## 编辑前

- 先阅读当前仓库的指令文件。
- 修改代码前检查相关项目上下文：
  - `pyproject.toml`、`uv.lock` 或依赖文件。
  - 现有包/模块结构。
  - 相关测试和 fixtures。
  - 现有的日志、配置和错误处理模式。
- 内容搜索使用 `rg <pattern>`，限定范围的文件发现使用 `rg --files -g '*.py'`（如果可用）。

## 实现

- 做满足请求的最小有效改动。
- 与本 skill 中的偏好设置有冲突时，优先遵循仓库现有风格。
- 除非用户要求或不重构无法安全完成更改，否则避免大范围重构。
- 没有具体需求时，不引入新依赖。
- 添加配置时，优先使用已有的配置系统。仅在项目没有更强约定时才使用 YAML。
- hooks 要实际可用、具体明确；不添加推测性的扩展点。
- 对于小型脚本或局部修复，按比例应用日志、注释和面向对象设计。
- 修改前先预览变更（如 `--diff`），确认后再应用——不盲信自动修复。

## 工具

- 项目使用 `uv` 时，优先用 `uv run ...` 运行 Python 命令。
- 可用或已配置时，优先用 `ruff check` 和 `ruff format`。ruff 迭代修复工作流见 `ruff-workflow.md`（仅当本次任务涉及 ruff 时，按入口路由读取）。
- 使用仓库声明的 Python 版本。仅在仓库没有更强版本约束时，新代码才默认用 Python 3.13+。
- 如果 `uv` 或 `ruff` 缺失，使用仓库现有命令，或跳过该检查并说明原因。
- 仅当任务真正依赖某工具且没有合理的本地替代方案时，才将其缺失视为阻断项。
- 除非用户授权或当前任务明确需要，否则不安装依赖。
- 传递目录作为参数（如 `ruff format src/`），不切换工作目录。

## 验证

- 根据改动的风险和范围，运行针对性的测试或检查。
- 代码改动时，优先运行针对性测试；当改动涉及共享模块时，再运行更大范围的测试套件。
- 仅在适合当前仓库和改动的情况下运行格式化或 linting。
- 如果跳过或无法运行验证，明确说明未运行的内容及原因。

## 最终自检

- 重读已改动的文件。
- 仅简化你改动的代码，或必须改动以满足请求的代码。
- 确认日志和注释有实际价值而非噪音。
- 确认错误信息包含足够的上下文以便诊断故障。
- 确认改动引入的性能风险已被规避，或已明确指出。
