# 项目背景

PyBits 是一个独立 Python 小工具集合。项目提供统一的 uv 环境和依赖管理，各工具相互独立、自包含。
最终需要将各个子项目包装为独立、完善、可执行的全局命令，而非仅在本项目运行项目指令。

## 项目结构

- `<TOOL_NAME>/`：各工具的独立目录。目录内 Python 文件使用 `snake_case.py`。
- `_shared/`：跨工具共享代码；`_shared/tests/` 存放共享结构和全局命令检查。
- `pyproject.toml`：统一依赖管理，脚本内不重复声明依赖。
- `.codex/`：Codex 的工作文件夹。
  - `.codex/_trash_bin_`：垃圾桶，用于存放软删除的文件。
  - `.codex/plans`：存放 Codex 的计划文档。
  - `.codex/logs`：存放 Codex 的执行日志。

# 最高优先级规则

以下规则优先级最高，必须始终遵守：

1. **严禁执行不可逆删除**；需要清理、覆盖或移除文件时，只能移动到 `.codex/_trash_bin_/`，并保留时间戳或原因标记。
2. 若发现信息不足、仓库状态冲突、引用/证据不足、验证失败或用户要求与代码事实不一致，先说明问题和影响，再继续推进。
3. 未经用户授权，不直接修改 `AGENTS.md`、`CLAUDE.md`、`.codex/settings.json`、`pyproject.toml`、`.python-version`或用户级/全局配置。
4. 涉及时效性事实、第三方 API/工具行为、依赖能力、外部文档或网页内容时，先验证再输出；网页检索优先使用 `$smart-search-cli`。
5. 涉及功能范围、命令名、输入输出契约、依赖引入或替换、架构边界、配置格式、数据处理、发布方式和历史改写时，Codex 只提供建议，等待用户确认后执行。
6. 不主动压缩、截断或概括用户要求完整保留的内容；需要缩写长输出时先说明范围和依据。


# 全局命令开发流程

- 每个工具最终都要通过 `pyproject.toml` 的 `[project.scripts]` 注册为全局命令，目标是在任意目录直接运行 `<TOOL_NAME>`。
- 新工具或重大改动前，先确认命令名、输入输出、依赖清单、测试样例，以及是否需要配置文件。
- 校验必须包含项目外目录调用，至少覆盖 `--help` 和核心路径；失败时说明测试目录、命令和错误。



# 工作流程

- 先读取用户点名文件、`README.md`、`pyproject.toml`、相关工具目录、共享测试，以及 `.codex/plans/`、`.codex/logs/` 中直接相关的最近记录；不要先写通用方案。
- 轻量问答和小范围文档/代码修改可直接处理；新工具、重大改动、跨多文件联动、规则文件修改、依赖变化或核心配置变化，先写 `.codex/plans/{stem}.md` 并等待用户确认。
- Python 代码任务使用 `python-engineering`；需求不清或会改变领域/术语/功能边界时可使用 `grill-with-docs`，但能从仓库文件推断的问题先查文件。
- 新工具或全局命令改动必须核对命令名、入口、输入输出、依赖清单、配置需求和测试样例；每个工具最终应在 `pyproject.toml` 的 `[project.scripts]` 注册。
- 执行时优先最小改动，复用现有目录、命名和共享工具；不顺手重构、迁移或美化无关文件。
- 修改入口、依赖或全局命令后，按风险运行匹配验证；常用检查包括 `uv run python _shared/tests/basic_info_check.py`、`uv run python _shared/tests/global_command_check.py`、相关工具的 `uv run <COMMAND> --help`，必要时再刷新全局安装。
- 全局命令校验必须覆盖项目外目录调用；失败时说明测试目录、命令和错误。
- 非轻量任务结束后，把修改对象、内容摘要、验证结果和遗留问题记录到 `.codex/logs/{stem}.md`。
