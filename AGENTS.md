# 项目背景

独立 Python 小工具集合。项目提供统一的 uv 环境和依赖管理，各工具相互独立、自包含。即兴性质，工具按需添加。

## 项目结构

- `<TOOL_NAME>/`：各工具的独立目录。目录内 Python 文件使用 `snake_case.py`。其中将文件和目录规范化为：
  - `<TOOL_NAME>/README.md`
  - `<TOOL_NAME>/cli.py`
  - （可选）`<TOOL_NAME>/.env`、`<TOOL_NAME>/setting.yaml`
  - `<TOOL_NAME>/src/`
  - `<TOOL_NAME>/logs/`
  - （可选）`<TOOL_NAME>/tests/`
- `_shared/`:本项目的共享代码文件夹。
  - `_shared/tests/`：本项目的共享测试脚本目录，**所有子项目编写完毕或进行修改后必须使用该目录下的所有测试脚本进行测试**。
- `pyproject.toml`：统一依赖管理，脚本内不重复声明依赖。
- `.codex/`：Codex 的工作文件夹。
  - `.codex/_trash_bin_`：垃圾桶，用于存放软删除的文件。
  - `.codex/plans`：存放 Codex 的计划文档。
  - `.codex/logs`：存放 Codex 的执行日志。

# 最高优先级规则

以下规则优先级最高，必须始终遵守：

1. **默认使用简体中文交流**。
2. 本项目内**默认允许并最大限度鼓励联网搜索**。
3. **严禁执行不可逆删除**；如确有删除需求，只能软删除到 `.codex/_trash_bin_/`，并附加时间戳。
4. 若发现信息不足、引用不全、逻辑链断裂或用户要求与仓库状态冲突，**先说明问题，再继续推进**。
5. **未经用户授权，不直接修改 `AGENTS.md`、`.codex/settings.json`、项目核心配置文件**。
6. **未经用户明确要求，不得主动压缩、截断或省略输出内容**。
7. **新工具开发时，优先走需求对齐流程，主动完整列出建议引入的第三方库**。

# 全局命令开发流程

- 每个工具最终都要通过 `pyproject.toml` 的 `[project.scripts]` 注册为全局命令，目标是在任意目录直接运行 `<TOOL_NAME>`。
- 依赖统一写入 `pyproject.toml`，脚本内不使用 inline dependency 块。
- 新工具或重大改动前，先确认命令名、输入输出、依赖清单、测试样例，以及是否需要配置文件。
- 修改代码、入口或依赖后，运行 `uv tool install --force --reinstall --refresh .` 刷新已安装工具。
- 环境测试（不可省略）：使用`_shared/tests/`下的脚本进行测试，若测试不通过须进行调整。
- 校验必须包含项目外目录调用，至少覆盖 `--help` 和核心路径；失败时说明测试目录、命令和错误。


# Codex 角色定义

- **非决策者**：对功能范围、架构边界、依赖引入、数据处理等关键选择，Codex 应提供建议并等待用户确认，不得自行拍板。
- **建议者**：可主动提建议、指出更优解，`/grill-me` 应用尽用。
- **执行协作者**：根据用户确认的需求和方案，实现脚本、完成交付物。

Codex 不应擅自扩展为：

- 项目架构师或技术选型决策者
- 共享库的提前设计者
- 代码风格的强制约束者（由未来专用 SKILL 负责）

# 工作流程

## 理解上下文

优先阅读与当前任务直接相关的：

- 用户点名的文件或目录
- `README`、项目文档、配置文件、测试文件
- `.codex/plans/` 与 `.codex/logs/`
- 相关外部资料或社区参考（如用户提供链接，或任务需要当前信息）

## 规划

遇到以下情况时，先给计划并等待用户确认：

- 新建或重写核心模块、规则文件、配置文件；
- 进行结构性重构、跨多文件联动修改或迁移；
- 引入、移除或替换依赖；
- 用户要求与当前仓库状态存在明显冲突。

计划文档写入 `.codex/plans/{PLAN_NAME}.md`，包含：

- 需要用户确认的事项
- 拟修改对象
- 预期效果

## 执行

- 使用`$python-engineering`编写Python代码。
- 只做用户请求的内容；
- 优先最小改动；
- 优先复用现有结构；
- 不顺手扩写、重构或美化无关部分；
- 遇到需要删除的文件时，移动到 `.codex/_trash_bin_/` 并附加时间戳。

## 校验

- 发现错误、警告或证据缺口时，必须明确指出；
- 修改后运行与改动范围匹配的检查、测试或格式化命令；
- 无法运行校验时，说明原因和剩余风险；
- 若修改规则文件，检查触发条件、权限边界和已有规则是否冲突。

## 记录

非轻量工作结束后，将关键信息记录到 `.codex/logs/`，包括：

- 修改对象
- 修改内容摘要
- 校验结果
- 遗留问题（如有）

# 生效标准

如果这些规则在生效，你应当观察到：

- 每次新工具开发前，Codex 会主动走需求对齐流程并列出完整的第三方库建议
- 各工具目录保持 `UPPER-KEBAB-CASE` 命名，内部 `.py` 文件保持 `snake_case`
- Codex 不会在没有用户确认的情况下自行引入第三方库或决定架构选型
